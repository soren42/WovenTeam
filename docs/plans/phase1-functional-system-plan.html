Phase 1 Goal

Phase 1 should produce a system that functions at some level as a real collaborative work platform. A human operator can create an initiative, the system can route work to suitable model-backed agents, at least two primary model paths can produce useful artifacts through controlled adapters, and the web console can show status, outputs, and operator decisions.

Definition Of Done

Phase 1 is done when this operator flow works without hand-editing ledgers or reading raw logs:

Open the web console and create an initiative or task package.
Route the task by role, agent availability, and declared tool policy.
Run a controlled adapter in a per-task workspace.
Record running, complete, failed, blocked, retry, and closed task states.
Open task detail in the web UI and inspect events, artifacts, and result text.
Approve, reject, retry, cancel, or close from the operator surface.
Display token allocation and actual usage where adapters report it.
Restart services and recover the same task state.

Workstreams

1. Durable Task State

Launch Critical

Keep JSONL as the audit log, but add a queryable state projection so the UI, CLI, and daemon have one coherent task view.

  • Add SQLite projection for tasks, events, artifacts, usage, and dependencies.
  • Rebuild projection from JSONL on daemon startup.
  • Add task detail, status, retry, cancel, close, and reopen APIs.
  • Preserve import/export and append-only recovery behavior.

2. Adapter Execution Plane

Launch Critical

Turn room participants into useful workers through explicit adapter contracts and controlled per-task workspaces.

  • Formalize adapter input, timeout, workspace, artifact, and result event contracts.
  • Harden the Codex path for bounded repo and documentation work.
  • Add Claude or Gemini as the second useful work path.
  • Record adapter command metadata, output files, result summary, and failure class.

3. Assignment And Routing

Core System

Move beyond manual agent choice with a small routing layer that respects role, availability, capability, policy, and blocked vendors.

  • Add role-to-agent routing config and capability registry.
  • Route by task role, tool profile, model availability, and active workload.
  • Preserve manager subtask parentage and dependency blocking.
  • Add basic spawn caps per initiative, parent task, and agent.

4. Web UI Operator Controls

Launch Critical

Make the dashboard the primary control plane rather than a monitor that requires CLI fallback for ordinary work.

  • Add task detail panel with event timeline and artifact viewer.
  • Add retry, cancel, close, approve, reject, and request revision controls.
  • Expand configuration UI for adapter enablement, routing, and health.
  • Keep unimplemented surfaces visibly disabled and honest.

5. Token, Cost, And Quota Control

Core System

Build on current allocation telemetry and add real usage where adapters expose it.

  • Distinguish allocated tokens from reported actual usage.
  • Record task usage events with provider, model, input, output, total tokens, and estimated cost.
  • Add warning and hard-stop behavior for task, initiative, daily, and monthly budgets.
  • Show budget pressure before task launch.

6. Launch Operations

Operations

Make the system maintainable as a live intranet service.

  • Expand health checks for daemon, ledger, projection, adapters, and service age.
  • Add backup and restore scripts for config, ledgers, database, and runtime artifacts.
  • Document log rotation, systemd recovery, and common failure diagnosis.
  • Run a launch rehearsal and record results.

Proposed Sprints

Sprint 1: State And Detail

Give every later feature a stable state model.

  1. SQLite projection from JSONL.
  2. Startup reconciliation.
  3. Task detail API.
  4. Task detail panel.
  5. Retry, cancel, close CLI/API skeleton.

Sprint 2: Execution Plane

Make useful work happen through adapters.

  1. Adapter contract.
  2. Codex adapter hardening.
  3. Claude or Gemini adapter path.
  4. Artifact and result viewer.
  5. Capability registry.

Sprint 3: Routing And Gates

Add managed collaboration controls.

  1. Role routing config.
  2. Capability-aware assignment.
  3. Review gate controls.
  4. UI retry, cancel, close.
  5. Spawn and capacity counters.

Sprint 4: Quotas And Operations

Prepare for regular use.

  1. Actual usage events where available.
  2. Budget enforcement.
  3. Health, backup, restore tooling.
  4. Operator runbook.
  5. Launch rehearsal.

Target Architecture For Phase 1

Phase 1 should keep the Phase 0 simplicity where it helps: append-only audit first, explicit adapters, and no hidden autonomous execution outside the task policy.
