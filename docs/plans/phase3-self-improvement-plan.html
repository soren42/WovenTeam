Phase 3 Goal

Phase 2 produced a functional workbench that the operator could trust for bounded real work. Phase 3 makes that workbench safe enough, observant enough, and autonomous enough to ship WovenTeam's own next feature, using WovenTeam — with the operator supervising at a glance rather than per task. The phase ends with v1.0: tagged, documented, and ready for first release.

Definition Of Done (v1.0)

Phase 3 ships when this full launch flow works without operator hand-holding. v1.0 has two gates: a product gate that proves WovenTeam can ship its own next feature, and a release gate that proves a second operator can install + recover + revoke + reason about the threat model.

Agents emit heartbeats on a configurable interval (default ~2 h) plus milestone events at every major lifecycle transition.
Long-running adapter runs renew their lease before expiry; cancel terminates the adapter process group with audit evidence.
Audit export supports windowed pagination so the 1 MiB single-buffer no longer caps initiative size.
A status report endpoint aggregates initiatives, in-flight tasks, alerts, budgets, autonomy grants, and heartbeats into one structured payload; the console shows the same data live via SSE.
Operator can grant a task a bounded, revocable autonomy grant (scope, TTL, wall-clock cap, network class, credential class) and the agent respects it without re-prompting. Revocation is immediate.

Workstreams

Use the filter to narrow to a category. Streams cluster the discrete pieces of work, each of which lands as one or two sprints below.

1. Autonomy & Permissive Modes

Launch Critical

Make it possible to grant an agent enough latitude to do real work without per-action approvals, while keeping every elevated action auditable and reversible.

  • Per-task autonomyLevel field (observe, ask-each, ask-batch, autonomous) with sane defaults per toolPolicy.profile.
  • Adapter plumbing: pass-through of --dangerously-skip-permissions for Claude Code, --ask-for-approval never for Codex CLI, equivalent flags for Gemini CLI, all gated by autonomyLevel.
  • Per-agent default autonomy (config), per-initiative override, per-task override. Most-specific wins.
  • Every elevated action records a woventeam.autonomy_event.v0.1 ledger entry: actor, target, command-class, and reason for granting.
  • Escalation path: a task may downgrade itself ("I need to write outside the workspace") and prompt the operator; the policy evaluator rejects unauthorized escalations.

2. Heartbeats & Milestone Reporting

Launch Critical

Surface long-running work to the operator without polling and without forcing each adapter to invent its own status protocol.

  • Each wt-agent emits woventeam.heartbeat.v0.1 records on a configurable interval (default 2 h). Heartbeats carry agent, host, last-lease-renewal, current task, and a free-text status line.
  • Milestone events: a fixed set of named transitions (received, plan, execute, review, complete, escalate) recorded as woventeam.milestone.v0.1 task events.
  • Aggregated GET /api/status: initiatives, in-flight tasks, recent heartbeats, alerts, budget pressure, and adapter readiness in one structured payload.
  • Notification routing: each milestone class can map to a Slack webhook key or email address; channels keyed off the existing slack_webhook.txt format.
  • Console status board: a single-pane summary of fleet-wide work, with explicit "last seen" stamps so an unresponsive agent is immediately visible.

3. Deliverables Pipeline

Launch Critical

Define where finished work lives. Today, accepted artifacts stay in the per-task runtime workspace; v1.0 needs a project-level destination.

  • Per-initiative deliverableRoot setting (defaults under data/deliverables/<initiativeId>/).
  • Promote-to-deliverable lifecycle that extends the artifact accepted state: file is copied (or moved) to the deliverable root and recorded as woventeam.deliverable.v0.1 with size, sha256, source workspace path, reviewer, and timestamp.
  • Optional packaging: tarball (zipped under data/deliverables/), branch (commit to a deliverables branch in the project repo), or pull-request (open a PR via gh using the existing project credentials).
  • Audit export folds the deliverable manifest into /api/initiative-audit output.
  • Operator console action: "Send to deliverable" button on accepted artifacts, with a dropdown for the packaging mode.

4. Remote Execution Substrate

Core System

Let agents run on more than one host without sacrificing audit, policy, or recovery semantics.

  • Two transports: SSH-launched wt-agent (simpler, reuses operator credentials) and HTTPS-polled wt-agent --remote (longer-lived, mTLS-authenticated).
  • Host capability registry: each host advertises which adapters + tool profiles it supports; the policy evaluator already has a hook for capability-based routing.
  • Per-task executionHost field. Routing picks a host that matches both the role and the available adapter.
  • Auth: bearer-token + IP allow-list for the local network on the daemon side; mutual TLS for cross-network use.
  • Failure modes: partition tolerance via lease expiry already there; new woventeam.host_event.v0.1 covers host up/down notifications.
  • Tested against at least one second host (real machine or container) before launch.

5. Real-Time Dashboard

Console

Operator can see what is happening and steer it in real time, not through poll-refresh.

  • SSE channel for every task / initiative / artifact / policy / heartbeat event so the console stops polling. Existing /events is the foundation.
  • Glanceable header summary: one row showing initiatives in flight, tasks active, alerts, budget pressure, oldest stuck task.
  • In-flight task controls: modify priority / agent / model, cancel, reroute, supply a one-shot operator note that lands as a task event.
  • Hotkey triage: a small set of keyboard shortcuts to acknowledge alerts, promote artifacts, reclaim leases, all surfaced in the help overlay.
  • Drag-priority queue for the assignment backlog: the dashboard updates the per-task priority field via a new POST /api/task-priority endpoint.

6. Self-Improvement Demonstration

Release

Drive a real WovenTeam feature initiative through WovenTeam itself. Capture the audit story end-to-end.

  • Choose a small but real feature that is not already on the roadmap (e.g. an exporter, a new adapter flag, a documentation generator).
  • Open the initiative as a CEO task; let the Project Manager role decompose into adapter-backed subtasks.
  • Real adapter execution under autonomous-with-audit mode; the operator only intervenes on policy denials or stuck leases.
  • Promote the result through the deliverable pipeline to a branch; open a PR; merge it; cite the audit JSON in the PR description.
  • Document the demonstration in the v1.0 launch report.

7. Release Hardening & v1.0

Release

Make the system installable and operable by someone who did not build it.

  • Public README pass, runbook polish, cheat sheet polish, screenshots.
  • Install + upgrade scripts (make install, distribution tarball, optional Debian packaging).
  • Branding pass on the console (logo, name, hue swatches).
  • Security review: review threat model (single tenant intranet) and document operational assumptions.
  • Tag v1.0, build a signed (or hashed) release artifact, archive a checksums file under docs/launch/.

Proposed Sprints

Seven sprints, resequenced from the original draft after a Codex review (see /woventeam/docs/audit/phase3-plan-peer-review-2026-05-20.md). Observability and control-plane work land first so autonomy is built on top of revocable, observable infrastructure rather than the other way around.

Sprint 1: Runtime Observability & Control Plane

Build the substrate autonomy will sit on: heartbeats, milestones, lease renewal, kill semantics, audit windowing.

  1. wt-agent emits woventeam.heartbeat.v0.1 on a configurable interval (default 2 h, override via heartbeatIntervalSeconds).
  2. Milestone events: woventeam.milestone.v0.1 at plan, execute, review, escalate, complete.
  3. GET /api/status returns one aggregated payload: initiatives, in-flight tasks, recent heartbeats, alerts, budgets, adapter readiness, host roster.
  4. Lease renewal: long-running adapters refresh their lease before the 15-minute expiry (deferred from Phase 2). Renewal events appear in the ledger.
  5. Kill semantics: POST /api/task-cancel terminates the adapter process group with SIGTERM then SIGKILL; appends a woventeam.kill_event.v0.1 record with exit reason. Cancelled output cannot be later accepted.
  6. Audit pagination: /api/initiative-audit accepts ?since=<ms> and ?limit=<N>; the 1 MiB buffer becomes a windowed stream (deferred watch item from Phase 2 launch report).
  7. Notification routing keyed off the existing webhook map plus optional smtp.conf. Defaults: escalate + stuck broadcast; routine milestones silent.
  8. Console status board panel renders the same data live via SSE.
Acceptance: integration test confirms heartbeat appends, status endpoint aggregation, lease renewal across a 20-minute synthetic run, cancel terminates the process group with audit evidence, paginated audit responses match the unpaginated baseline, and Slack webhook delivery hits a local mock. Console status board renders within 2 s of a heartbeat append.

Sprint 2: Autonomy & Policy Contract

Now that the control plane exists, grant agents bounded, revocable latitude to do real work.

  1. Add autonomyLevel to the task package schema; defaults derived from toolPolicy.profile.
  2. New autonomyGrant sub-object on every task with scope (workspace paths, allowed endpoints, allowed git operations), ttlSeconds, maxWallClockSeconds, maxCostUsd / maxTokens (inherited from budget), network (none/intranet/allowlist), credentialClass (none/repo-read/repo-write/release), requiresCleanWorktree, revokedAtUnixMs / revokedBy.
  3. Adapter plumbing: pass-through of CLI permission flags (--dangerously-skip-permissions, --ask-for-approval never, etc.) only when grant.ttl is unexpired AND scope covers the invocation.
  4. Per-agent default config; per-initiative + per-task overrides; most-specific wins.
  5. New woventeam.autonomy_event.v0.1 ledger record for every elevation and every adapter invocation. The plan does not claim to observe every filesystem write the adapter performs once skip-permissions is granted (OS-level tracing is out of scope for v1.0); the audit records the invocation envelope and the CLI exit code, not each syscall.
  6. Policy evaluator gains autonomy_required reason so unauthorized elevations fail closed; autonomy_expired and autonomy_revoked as separate codes.
  7. Revocation endpoint: POST /api/autonomy-revoke with taskId + reason. Revocation is immediate; the next adapter invocation downgrades to the lowest level that still fits the tool profile.
Acceptance: integration test (wt-autonomy) creates a task with autonomy autonomous and an explicit grant, runs the adapter with the corresponding skip-permissions flag, and verifies the autonomy_event chain. Mid-task autonomy-revoke downgrades the next adapter invocation. Expired grants fail closed with autonomy_expired. Grants without sufficient scope are denied with autonomy_required.

Sprint 3: Deliverables & Provenance

Accepted work product lands in a stable, scanned destination.

  1. Schema: woventeam.deliverable.v0.1 with source workspace path, deliverable path, size, sha256, reviewer, packaging mode.
  2. POST /api/task-deliverable + wt-task artifact ship TASK_ID [--mode copy|tarball|branch|pr].
  3. Per-initiative deliverableRoot config (default data/deliverables/<initiativeId>/).
  4. Optional packaging: copy (default), tarball (zip), branch (commit + push), pull-request (uses local gh auth, operator-gated by --yes).
  5. Pre-ship secret scan: branch + pr modes refuse to ship if the staged content matches a small set of token / private-key / API-key patterns. The scan result lands as a woventeam.secret_scan.v0.1 record.
  6. Rollback / supersede: a new deliverable can supersede an earlier one; the manifest preserves the predecessor's sha256 + timestamp so the audit trail remains coherent.
  7. Audit export embeds the deliverable manifest; the initiative card surfaces a "Deliverables" sub-panel.
Acceptance: integration test promotes an artifact through tarball + branch modes; manifest checksums match; the secret scan rejects a planted token; audit export includes the deliverable + secret_scan records. PR mode requires explicit --yes and is exercised only against a local fixture repo in test.

Sprint 4: Remote Execution Substrate

Agents on additional hosts claim work with the same audit + policy + revocation semantics.

  1. Two transports: wt-agent --remote (HTTPS poll + bearer token) and wt-agent --ssh HOST (operator-launched).
  2. Bearer-token auth + IP allow-list on the daemon. POST /api/auth-token rotates a token; POST /api/auth-token/revoke burns one. Tokens carry an issued-at and TTL.
  3. Host capability registry: POST /api/host-capabilities from the agent, GET /api/hosts for the operator. Capability changes broadcast as woventeam.host_event.v0.1.
  4. Routing extension: tasks with executionHost route only to that host; capability check rejects mismatches with capability_unmet.
  5. Operator auth on mutating endpoints: task-priority, task-reroute, task-cancel, autonomy-revoke, auth-token, task-deliverable all require either a local-socket origin or a bearer token issued to "operator" role. The threat-model assumptions land in docs/security/threat-model.html.
  6. Partitions surface as stuck-lease alerts and reclaim — the Phase 2 reclaim path generalizes to remote hosts unchanged.
Acceptance: integration test runs two containerized agents on different network namespaces against one daemon; both successfully lease + complete tasks; partitioning one host triggers reclaim within the lease window; token revocation invalidates an in-flight remote agent's next claim with token_revoked; mutating endpoints reject anonymous requests with classified reason codes.

Sprint 5: Real-Time Dashboard

Operator sees and steers in-flight work in real time.

  1. SSE channels for tasks, initiatives, artifacts, policy decisions, autonomy events, heartbeats, host events; replace polling in web/app.js.
  2. Glanceable header summary with one-line status: initiatives, in-flight tasks, alerts, budget pressure, oldest stuck task, autonomy grants in flight.
  3. In-flight controls: priority change, agent reroute, cancel (kill-tree), operator note, autonomy revoke. New POST /api/task-priority + POST /api/task-reroute endpoints; cancel + revoke reuse Sprint 1/2 endpoints.
  4. Hotkey triage: A acknowledge alert, P promote selected artifact, R reclaim selected task, X cancel selected task, ! revoke autonomy on selected task.
  5. Drag-priority queue for the task backlog with optimistic UI + server reconciliation.
Acceptance: manual smoke against a running daemon shows task state changes propagate within 2 s of the ledger append. New endpoints are covered by integration tests. Help overlay maps every hotkey to a CLI equivalent.

Sprint 6: Release Hardening Rehearsal

Prove WovenTeam can be installed, recovered, and operated by a second pair of hands before the self-improvement demo runs.

  1. Backup / replay / restore: delete the SQLite projection, replay data/task-packages.jsonl, verify status/audit/deliverables match a captured baseline byte-for-byte (modulo timestamps).
  2. Install + upgrade rehearsal: a second operator (or a freshly provisioned VM) installs WovenTeam from a tarball using only the README, starts the daemon, runs make test, and exercises the cheat sheet's walkthrough.
  3. Failure injection: daemon restart, agent restart, remote partition, expired token, full disk on the deliverable root, corrupted JSONL line, oversized audit response. Each failure mode maps to a classified ledger event and a documented recovery path.
  4. Threat-model signoff: docs/security/threat-model.html documents the single-tenant intranet assumption, the mTLS bar for cross-network, the operator-auth assumption on mutating endpoints, and the explicit out-of-scope items.
  5. Secret-handling pass: redaction at every emission point (logs, artifacts, heartbeats, adapter manifests, deliverable manifests). The pre-ship secret scan from Sprint 3 stays; this sprint adds the redaction-at-source step.
Acceptance: backup/replay/restore produces a matching baseline; the failure-injection scenarios each have a reproducible test or runbook step; second-operator install completes from tarball within 30 minutes following only the README. Threat-model doc reviewed.

  2. Open the initiative as a CEO task package; let PM decompose into subtasks with autonomy autonomous bounded by a scoped grant.

Target Architecture For Phase 3

Phase 3 keeps the Phase 0-2 storage discipline: append-only JSONL remains the recovery source, SQLite remains rebuildable state, and every new surface (autonomy, heartbeats, deliverables, host events) appends one new schema rather than mutating prior ones.

Autonomy Levels & Permissive Modes

Autonomy is the operator's contract with an agent about how much it may do without re-prompting. Levels are deliberately discrete so audit and UX can both reason about them. Each task carries a bounded, revocable grant alongside the level — the level says "what kind of latitude", the grant says "for how long, over what scope, with what credentials."

LevelAdapter flagWhat the agent may do
observeread-only flagsRead repo, summarize, plan. No filesystem or network writes.
ask-eachdefault interactiveDefault for new tasks. Adapter prompts for every privileged action; operator approves inline or via console.
ask-batchscope-batched promptsApprove a batch of actions ahead of time (e.g. "all edits under src/foo"). Adapter executes inside the batch without re-prompting.
autonomous--dangerously-skip-permissions, --ask-for-approval never, equivalentsFull latitude inside the grant. Every elevation and every adapter invocation still appends an autonomy_event; the grant bounds blast radius.

Grant fields (required for autonomous; optional below)

  • scope — workspace paths, allowed endpoints, allowed git operations (e.g. branch-only, commit, push).
  • ttlSeconds, maxWallClockSeconds — the grant expires; the policy evaluator rejects elevations past either.
  • maxCostUsd, maxTokens — inherited from the task budget but separately enforced inside the grant.
  • network — none, intranet, or allowlist (with explicit hosts).
  • credentialClass — none, repo-read, repo-write, release. Controls which tokens the daemon will lease to the adapter.
  • requiresCleanWorktree — true for branch / pr deliverable modes.
  • revokedAtUnixMs, revokedBy — populated by POST /api/autonomy-revoke; revocation is immediate.

Resolution order

  1. Per-task autonomyLevel + autonomyGrant (most specific; operator's explicit choice).
  2. Per-initiative default (set at initiative creation).
  3. Per-agent config default (defaultAutonomyClaude, etc.).
  4. Profile default (e.g. observe tasks default to observe autonomy).

What the audit can (and cannot) prove

Each elevation and each adapter invocation appends an autonomy_event record with actor, command-class, target, and reason. Once a skip-permission flag is passed to an external CLI, WovenTeam does not observe individual filesystem writes the CLI makes. OS-level tracing is out of scope for v1.0. The grant bounds, the invocation envelope, and the CLI exit code are what the audit guarantees; per-write provenance is not.

Heartbeats & Milestone Reporting

Heartbeats and milestones answer two different questions. Heartbeats answer "is this agent still alive and working" — they fire on a clock. Milestones answer "where in the work am I" — they fire on transitions.

Heartbeat payload (proposed)

{
  "schema": "woventeam.heartbeat.v0.1",
  "agent": "claude",
  "host": "homelab.lan",
  "currentTaskId": "task_abc123",
  "currentLeaseExpiresAtUnixMs": 1779260000000,
  "statusLine": "Implementing exporter; tests still failing.",
  "createdAtUnixMs": 1779253500000
}

Milestone payload (proposed)

{
  "schema": "woventeam.milestone.v0.1",
  "taskId": "task_abc123",
  "milestone": "review",
  "message": "Ready for operator review. Tests pass; output in result.md.",
  "createdBy": "wt-agent@claude",
  "createdAtUnixMs": 1779253600000
}

Notification routing

Operator picks which milestones broadcast and to where. The map is config-driven:

notifications.milestone.escalate=slack:on-call
notifications.milestone.complete=slack:work
notifications.heartbeat.stuck=slack:on-call
notifications.policy.deny=slack:audit

Deliverables Pipeline

An accepted artifact today is a file inside a per-task workspace. v1.0 needs that file to land in a project-level destination with a manifest the operator can hand to compliance, a downstream pipeline, or a second WovenTeam instance.

States

accepted (today) → shipping (in-flight copy/packaging) → shipped (deliverable manifest written) → optional archived (older deliverable superseded).

Packaging modes

  • copy (default): file moved to data/deliverables/<initiativeId>/<taskId>/<name>.
  • tarball: a zipped bundle with the manifest and all associated workspace files.
  • branch: commits to a deliverable branch (configurable name) and pushes if remote is configured.
  • pull-request: opens a PR via local gh credentials. Operator-gated by --yes like wt-rehearse-live.

Manifest fields

schema, taskId, initiativeId, sourceWorkspacePath, deliverablePath, sha256, sizeBytes, packagingMode, reviewer, notes, createdBy, createdAtUnixMs.

Network Execution Topology

Single daemon, many agents. Local agents talk to the daemon over the loopback interface; remote agents authenticate with a bearer token (intranet) or mTLS (cross-network). The same lease + policy + audit machinery applies — only the transport changes.

Self-Improvement Demonstration

The acceptance moment for Phase 3 is WovenTeam shipping a real WovenTeam feature using its own workbench. The demo is small but real; the audit story is what matters.

Candidate features

  • Markdown export. A new bin/wt-ops-export that turns an initiative audit JSON into a Markdown brief. Touches CLI + docs but not the core daemon.
  • Vendor adapter flag. Add a configurable claudeMaxTurns setting plumbed into the adapter's invocation. Touches config + adapter + integration test.
  • Audit query filter. A ?since=<ms> filter on the audit endpoint so dashboards can fetch deltas. Touches projection + endpoint + UI.

Demonstration flow

  1. Operator opens a CEO task package via the console naming the chosen feature.
  2. Project Manager role decomposes it into design + implementation + test + docs subtasks.

  5. Accepted artifacts are shipped as a deliverable branch + PR.
  6. Audit export captures the entire run; the PR description cites it.
  7. Operator intervention is limited to policy approvals on the few escalations that the agent decides need it.

What counts as success

The merged PR is the artifact. Its commit history, audit export, and the launch report tell the story. The launch report links to each, and notes the moments where the operator did or did not need to intervene.
