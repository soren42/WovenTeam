Phase 2 Goal

Phase 2 should produce a usable workbench for bounded real work. The operator can start an initiative, assign or route work to Claude, Codex, and Gemini paths where enabled, inspect live state and artifacts, enforce policy and budgets, and recover cleanly after daemon or agent restarts.

Primary outcome1Functional workbench

Initiative, tasks, agents, artifacts, review, and recovery in one operator loop.

Model target3Primary model paths

Claude, ChatGPT/Codex, and Gemini contribute where configured and launchable.

Safety target0Silent failures

Misconfigured adapters, blocked models, and budget stops fail closed with visible reasons.

Definition Of Done

Phase 2 is done when this full workbench flow works without editing raw ledgers or manually chasing service logs:

Create an initiative from the web UI or CLI and see initiative-level status, budget, and open gates.
Break the initiative into role-assigned tasks and manager-created subtasks.
Run real adapter preflights for Claude, Codex, and Gemini and block unsafe or unavailable paths.
Lease, pause, resume, retry, and recover agent work without double-claiming tasks.
Promote reviewed artifacts from raw output to accepted initiative assets.
Enforce model policy, blocked vendors, capacity caps, and token budgets across CLI and web paths.
Export an initiative audit report containing task history, gates, usage, artifacts, and decisions.
Run a Phase 2 launch rehearsal that survives daemon and agent restart.

Workstreams

1. Real Adapter Hardening

Launch Critical

Move from deterministic fake adapter proof to safe, observable real CLI execution.

  • Add adapter preflight checks for command path, auth/session state, runtime root, timeout, and tool profile compatibility.
  • Define failure classes for missing CLI, auth failure, timeout, nonzero exit, empty result, and output truncation.
  • Add harmless real-CLI smoke tasks for enabled Claude, Codex, and Gemini paths.
  • Surface adapter readiness and last failure in the web UI.

2. Initiative And Workspace Model

Launch Critical

Promote initiative-level work from a loose task grouping into the operator's primary unit of control.

  • Add initiative projection, summaries, and detail APIs derived from task packages and events.
  • Add CLI commands for initiative create, list, show, export, and close.
  • Separate initiative artifacts from per-task runtime artifacts while preserving append-only history.
  • Add initiative selector, status summary, budget summary, and open gate panel to the console.

3. Agent Workload Control

Core System

Give agents enough autonomy to do useful work while keeping operator control and crash recovery predictable.

  • Add claim or lease semantics so tasks cannot be double-run after races or restarts.
  • Add per-agent pause, resume, concurrency, cooldown, max runtime, and stuck-task detection.
  • Record attempt counts, retry causes, and terminal failure reasons.
  • Expose agent operations from CLI and web without requiring systemd access for routine control.

4. Review And Artifact Promotion

Workbench UI

Turn raw adapter output into reviewed assets that can be accepted, rejected, superseded, exported, or used by later tasks.

  • Add artifact states: draft, reviewed, accepted, rejected, superseded.
  • Record reviewer identity, notes, and decisions as task events.
  • Add artifact export and promotion commands for CLI users.
  • Improve the console artifact viewer for result, stdout, stderr, manifest, and promoted asset inspection.

5. Policy, Audit, And Cost Controls

Policy

Make longer runs safe by enforcing model policy, budget limits, and auditable decision history everywhere work can enter the system.

  • Centralize model/vendor policy checks for routing, CLI, web config, and manager subtask creation.
  • Block removed vendors and incompatible model-agent combinations before task acceptance.
  • Enforce budget limits by initiative, agent, model family, day, and month.
  • Export a single initiative audit report with tasks, events, gates, usage, artifacts, and policy decisions.

6. Launch Rehearsal And Documentation

Operations

Close Phase 2 with repeatable evidence, not a narrative claim.

  • Add a Phase 2 end-to-end test target covering initiatives, real-adapter preflight, leasing, promotion, policy, and recovery.
  • Run one harmless live adapter rehearsal against installed CLIs with operator-approved token budgets.
  • Update the operations runbook, user cheat sheet, API docs, and console status surfaces.
  • Produce a one-page Phase 2 launch report in the inline HTML/CSS/JS report format.

Proposed Sprints

Sprint 1: Adapter Hardening

Make real CLI execution safe enough to enable deliberately.

  1. Adapter preflight API and CLI.
  2. Failure taxonomy and ledger events.
  3. Harmless live smoke task templates.
  4. Adapter readiness UI.
  5. Tests for unavailable and failing adapters.

Sprint 2: Initiatives

Make initiative state the workbench's organizing layer.

  1. Initiative projection and APIs.
  2. wt-task initiative commands.
  3. Initiative detail and selector UI.
  4. Initiative artifact layout.
  5. Initiative budget and gate summaries.

Sprint 3: Workload Control

Prevent duplicate work and expose operational control.

  1. Task claim or lease records.
  2. Agent pause and resume.
  3. Attempt counts and retry causes.
  4. Stuck-task detection.
  5. Agent operations panel.

Sprint 4: Artifact Promotion

Separate raw output from accepted work product.

  1. Artifact state events.
  2. Promotion and export CLI.
  3. Reviewer notes and decisions.
  4. Accepted asset inventory.
  5. Console artifact review improvements.

Sprint 5: Policy And Audit

Make model, budget, and compliance limits first-class.

  1. Central policy evaluator.
  2. Blocked vendor enforcement.
  3. Initiative and model-family budgets.
  4. Audit export command.
  5. Policy and budget UI warnings.

Sprint 6: Launch Rehearsal

Prove the system can run a bounded real project.

  1. Phase 2 E2E test target.
  2. Harmless live adapter rehearsal.
  3. Restart and recovery exercise.
  4. Runbook and cheat sheet updates.
  5. Phase 2 launch status report.

Target Architecture For Phase 2

Phase 2 keeps the Phase 1 storage discipline: append-only JSONL remains the recovery source, SQLite remains rebuildable state, and real adapter execution remains opt-in through explicit policy and preflight checks.
