Detailed Technical Status

Schema & Config

  • autonomyLevel: none | request | autonomous. autonomyGrant: scope, ttlSeconds, maxWallClockSeconds, network, credentialClass, requiresCleanWorktree, revokedAtUnixMs.
  • Config defaults: defaultAutonomyLevel plus per-agent claudeDefaultAutonomyLevel, chatgptDefaultAutonomyLevel, geminiDefaultAutonomyLevel.
  • Resolution order in wt-agent: per-task override > per-initiative > per-agent config > toolPolicy profile default.

Policy Evaluator

  • New stable denial reasons in wt_policy.c: autonomy_required (unknown level / missing grant for non-default), autonomy_expired (grant TTL exhausted), autonomy_revoked (operator revoked).
  • Denials feed the same wt_notify module Sprint 1 added — autonomy denials surface on the Slack webhook with the policy decision body.

Ledger Event

  • woventeam.autonomy_event.v0.1 emitted at four points: grant observation on claim, adapter invocation, elevated adapter invocation, adapter exit. wtTaskAppendAutonomyEvent() in wt_task_store.c.
  • Revocation appends both an autonomy_event (reason: revoke) and a woventeam.kill_event.v0.1 with reason:"autonomy-revoke" so the Sprint 1 wait loop tears the adapter down.

Endpoints + CLI

  • POST /api/autonomy-revoke in wt_roomd.c — body { taskId, reason }. Wires through to Sprint 1's kill-event semantics.
  • wt-task autonomy revoke TASK_ID [--reason TEXT] wraps the endpoint.
  • wt-task create / request gain the six autonomy flags listed above; values flow into the task package's autonomyGrant block.

Web UI Parity

  • Settings panel exposes the per-agent autonomy defaults so operators can see what's in woventeam.conf.
  • Task composer adds grant controls (level + TTL + scope + network + credential class + clean-worktree).
  • Task detail surfaces the grant block plus expiry countdown.
  • Revoke Autonomy action button on tasks with an active grant; calls /api/autonomy-revoke.

Adapter Wiring

  • Codex: elevated path adds --sandbox danger-full-access and --ask-for-approval never; non-elevated stays on workspace-write + on-request.
  • Claude / Gemini: equivalent skip-permission / non-interactive flags are appended only when the grant is valid at fork time.
  • Grant is re-checked at each adapter invocation; revocation between invocations downgrades the next call.

Review Notes

  • Revocation uses Sprint 1's kill-event substrate. If Sprint 1's wt_agent.c wait loop or cancel semantics change, rebase and re-run make test-autonomy test-agent-workload-control test-codex-adapter.
  • Grant scope is deliberately small for v1.0: workspace, adapter, or * authorizes elevated adapter invocation. It does not claim per-syscall or per-file observability after a vendor CLI receives skip-permission flags — the trust boundary is at fork time.
  • requiresCleanWorktree is enforced agent-side before elevation, not in the daemon admission check. The daemon may later run on a different host than the worker (Sprint 4 remote execution), and only the worker can observe its own tree.
  • The autonomy_event chain is granular enough to reconstruct elevated-vs-downgraded decisions post-hoc from the ledger alone — useful for incident review.
Trust-boundary caveat: once an adapter is forked with skip-permission flags, WovenTeam relinquishes per-action visibility into that adapter's tree. Autonomy bounding is a fork-time admission control, not a per-call sandbox.

Follow-ups

  • Fold autonomy_events into the combined audit export now that Sprint 1's ?since/?limit windowing has landed.
  • Replace static grant scope strings with a structured allowlist before Sprint 4's remote worker execution. Free-form strings invite drift once multiple hosts share a grant vocabulary.
  • Gate POST /api/autonomy-revoke behind operator auth in Sprint 4 (remote execution) — currently relies on local-trust like other endpoints.
  • Wire Sprint 1's escalate-milestone and stuck-lease notification hooks (callouts in the Sprint 1 follow-ups) alongside the new autonomy_required denial reason so escalations and admission failures land in the same channel.

No new schemas are breaking; autonomyLevel and autonomyGrant are optional and pre-existing ledgers project without autonomy_event rows. Adapter elevation is additive — the dangerous flags only appear when explicitly granted, so the default behavior is unchanged from Phase 2.
