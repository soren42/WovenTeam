The Sprint 7 feature: initiative audit exporter

Sprint 7's deliverable is wt-task initiative report INIT [--out FILE] — it renders the /api/initiative-audit JSON as a self-contained HTML report. It was chosen as the demo feature precisely because it dogfoods the rest of Phase 3: the report itself is an HTML deliverable (Sprint 3 pipeline + the html-over-markdown directive), and the demo can render a report of its own initiative.

What it renders

  • Summary cards (tasks, active, complete, failed, accepted, deliverables, max tokens)
  • Task table with role + agent + priority
  • Deliverables with SHA-256 + size + reviewer
  • Secret-scan results (clean / match pills)
  • Policy denials + a full event timeline

How it's built

  • Pure bash + jq — no new dependency (jq was already required)
  • Inline-CSS single file, no external assets
  • Every interpolated value is @html-escaped at render time
  • Defaults to stdout; --out writes a file

Security note

  • A task title containing <script> renders inert (escaped), verified by the integration test — the report can be opened safely even when the ledger contains hostile-looking data.

The self-improvement demonstration

WovenTeam tracked the exporter's own development as a real initiative through its control plane. The CEO opened the initiative and filed a program_manager task; the PGM delegated to a project_manager; the PM decomposed into three worker subtasks (two backend_dev, one tester) routed to claude, chatgpt, and gemini; each produced an accepted artifact; the feature shipped through the deliverable pipeline; and finally the new exporter rendered the audit of this very initiative.

The artifact: the rendered self-audit report is captured at docs/launch/phase3-v1.0-demo-report-2026-05-21.html — a report WovenTeam generated about its own v1.0 feature initiative, using the feature that initiative delivered. Five tasks, the full role hierarchy, the shipped deliverable with its SHA-256, a clean secret scan, and the complete event timeline.
Adapter-execution boundary (honest disclosure): the demonstration exercises the real control plane — ledger, projection, role-spawn policy (CEO→PGM→PM→workers), token budgets, the deliverable pipeline, and the exporter. The worker lifecycle was operator-driven rather than live model inference: the adapters (codex / claude / gemini CLIs) are present but left disabled so the demo is deterministic, free, and reproducible without credentials. A credentialed autonomous run would swap the operator-driven update-status/artifact promote steps for real adapter output under an autonomy grant; every other step is identical.

Post-v1.0 backlog

Items deferred through Phase 3, carried into the post-v1.0 roadmap:

  • Credentialed autonomous demo. Re-run the Sprint 7 demonstration with adapters enabled under a real autonomy grant, so the worker steps are live model inference rather than operator-driven.
  • Local operator transport. The operator session is a loopback-issued shared secret in a 0600 file ("local host = operator"). A Unix-domain socket or per-operator session isolation is the next step if multi-user local hosts come into scope.
  • Live-browser dashboard smoke. The Sprint 5 dashboard's drag-drop + hotkey ergonomics still want a real-browser click-through; automated coverage validates the API/SSE paths only.
  • Per-client SSE fan-out. Each /events client tails the ledger independently; a shared broadcast buffer would scale to many viewers.
  • Cross-namespace remote test. The remote-execution acceptance test runs over loopback where unprivileged netns is unavailable; a privileged run should repeat it across real network boundaries.
