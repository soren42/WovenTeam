Detailed Technical Status

Module Surface

  • include/wt_deliverable.h + src/wt_deliverable.c - new module. Top-level entry: wtDeliverableShip(config, request, result).
  • Independent helpers: wtDeliverableSha256, wtDeliverableScanFile (uses POSIX regex.h, REG_EXTENDED + REG_NOSUB).
  • External tools shelled out via a systemReliable() wrapper that temporarily restores SIGCHLD=SIG_DFL around the call. Daemon's SIG_IGN policy breaks bare system()/popen(); the wrapper resolves the ECHILD trap.

Config Knobs

  • deliverableRoot (default data/deliverables) - per-initiative directory created on first ship.
  • deliverableDefaultMode (default copy) - used when the request body omits a mode.
  • deliverableBranchPrefix (default deliverables) - branch + PR modes commit to <prefix>/<initiativeId>.
  • secretScanPatternsFile (default config/secret-scan-patterns.txt) - referenced file matches the slackWebhookFile pattern; KEY=REGEX, one per line.

Endpoint + CLI

  • POST /api/task-deliverable - body { taskId, mode, reviewer, supersedes, repoPath, yes, prTitle, prBody, destBasename }. Classifies failures into stable reasons: secret_scan_block, missing_yes_gate, repo_path_missing, task_not_accepted, source_missing.
  • wt-task artifact ship TASK_ID [--mode copy|tarball|branch|pr] [--reviewer NAME] [--supersedes ID] [--repo PATH] [--yes] [--pr-title T] [--pr-body B] [--dest-basename NAME] wraps the endpoint.

Projection + Audit

  • Two new SQLite tables: deliverables (per-ship row, raw JSON cached) and secret_scans (zero-or-one per ship).
  • Projection rebuild DROPs both before re-running so the SQLite is rebuildable from the ledger alone.
  • GET /api/initiative-audit emits two new arrays - deliverables[] and secretScans[] - alongside the existing events/policyDecisions/usage. Both honor the existing ?since+?limit cursor.
  • New projection helper wtTaskProjectionResolveAcceptedArtifact(taskId) returns the task's initiative + accepted artifact path so the endpoint resolves the source workspace without parsing the audit JSON.

Web UI

  • Task detail panel gains a "Send to deliverable" strip when the focused task has an accepted artifact. Mode dropdown drives field visibility (repo path appears for branch/PR; explicit-yes checkbox for PR only).
  • New Deliverables sub-panel sits below the Promoted-assets panel. One row per ship, with a colour-coded scan pill (clean = green, hit = red) per row.
  • Browser JS only - no new build dependency.

Sprint 2 Integration Notes

Per the alternating-models cadence, Sprint 3 was built on top of Sprint 2's autonomy substrate. A read-pass of the autonomy surface surfaced two integration considerations:

  • Trust boundary. Shipping a deliverable is an operator action, not adapter execution, so it bypasses Sprint 2's grant resolution and adapter elevation. The endpoint relies on local-trust (same as the existing artifact-promotion endpoint) and Sprint 4 will gate it behind operator auth alongside autonomy-revoke, task-cancel, task-priority, etc.
  • Provenance. The deliverable schema reserves an autonomyProvenance field for recording whether the originating task ran under an autonomous grant. The endpoint accepts autonomyLevel + autonomyElevated in the body so a caller (typically the future autonomous workflow) can record provenance at ship time. The endpoint does not walk the ledger to look it up on every ship; that would be an O(N) re-scan we can defer until Sprint 4's structured allowlist tightens the autonomy grant model.

No changes were needed to Sprint 1 or Sprint 2 code; the ledger, projection, and notify pipeline absorbed two new schemas additively.

Follow-ups

  • Sprint 4 operator-auth gate. /api/task-deliverable joins the list of mutating endpoints that must require either a local-socket origin or a bearer token issued to operator role. Documented in the rev 2 plan; this sprint did not implement auth.
  • Autonomy-provenance lookup helper. Add a bounded ledger-tail scan (read the last ~1 MiB) so deliverable manifests can record autonomy state without an explicit caller-supplied field. Useful for Sprint 7's self-improvement demo where the autonomous flow ships its own deliverable.
  • Structured grant scope. The grant scope is still free-form strings (workspace/adapter/*); Sprint 4 will introduce a structured allowlist before remote-host expansion. The deliverable schema is already prepared for that move - autonomyProvenance takes a level + boolean now and can grow without a v0.2 schema.
  • Streaming secret scan. The current implementation reads up to 4 MiB into memory; for large deliverables (binary builds, screenshots, datasets) this should switch to chunked scanning. Defer until a real consumer needs it.
  • Redaction on shipping. Today, copy/tarball records a scan hit but doesn't redact. Sprint 6's secret-handling pass will add an opt-in redaction mode that rewrites matched substrings before shipping.

All schema additions are additive; old ledgers without deliverable/secret_scan events project as empty tables. The new systemReliable() wrapper resolves a long-standing daemon SIGCHLD compatibility issue and is available for future modules that need to invoke shell tools.
