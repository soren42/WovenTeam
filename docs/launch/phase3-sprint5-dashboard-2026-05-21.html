Detailed Technical Status

Server: SSE Ledger Delta

  • streamLedgerDelta() tails task-packages.jsonl by byte offset and emits one event: ledger frame per new complete line, forwarding the full JSONL record as the SSE data.
  • Seeded to the file's current size at connect time (0 if absent) so only post-connect records stream — no history replay. Truncation/rotation resyncs to EOF.
  • Runs inside the existing 500 ms handleEvents tick alongside the chat message stream; heartbeats only fire when neither stream had output.

Client: Event-Driven Refresh

  • connectEvents() adds a ledger listener; dispatchLedgerEvent() maps schema → debounced loader (task_event → tasks + header + initiatives; heartbeat → status board; host_event → hosts; deliverable → deliverables; task_usage → tokens; policy_decision → header).
  • scheduleRefresh() collapses event bursts into one refresh per loader.
  • Polling intervals dropped from 5-15 s to a 30 s fallback (adapters 60 s).

In-Flight Controls

  • POST /api/task-priority + /api/task-reroute (landed in Sprint 4 for the auth gate) now have UI; POST /api/task-note is new this sprint.
  • Projection gap fixed: the priority event type now updates the task's priority column (Sprint 4 appended the record but the projection ignored it).
  • All three are gated by requireOperatorAuth; the loopback console passes via the peerIsLocal bypass.

Hotkeys + Help

  • A acknowledge oldest alert (operator-local), P promote selected artifact, R reclaim, X kill-tree cancel (/api/task-cancel), ! revoke autonomy.
  • Suppressed while typing in an input/select/textarea.
  • Help overlay extended with the new keys + a UI↔CLI parity row for each.

Drag-Priority Queue

  • Four lanes (urgent/high/normal/low) of draggable task chips; dropping a chip in a lane POSTs /api/task-priority with that tier.
  • Optimistic: the chip moves immediately; a failed POST triggers loadTasks() to reconcile from the server.
  • Native HTML5 drag events, no library (per the no-Node directive).

Sprint 4 Truncation Fixes

  • Cleared two latent -Wformat-truncation warnings in Sprint 4 code that surfaced on a clean build: the remote-claim response buffer (4096 → body-sized; could emit malformed JSON on a max-size task body) and the bearer-token extraction buffers (256 → 512 to match the auth header).

Sprint 4 Integration Notes

Per the alternating-models cadence, Sprint 5 was built on Codex's Sprint 4 substrate. The read-pass surfaced three things worth recording:

  • Endpoints already existed. /api/task-priority and /api/task-reroute were added in Sprint 4 to satisfy the operator-auth gate list. Sprint 5 found the priority endpoint appended a record the projection never applied, and fixed the projection — a latent no-op the auth-gate work didn't exercise.
  • Operator-auth respected. The new /api/task-note joins the operator-gated set. The browser console reaches all of them via the peerIsLocal loopback bypass, so no token plumbing was needed in the dashboard.
  • Clean-build hygiene. A clean build surfaced two -Wformat-truncation warnings from Sprint 4 (remote-claim response, token extraction) that incremental builds had hidden. Both fixed; the remote-claim one could have produced malformed JSON on a maximum-size task body.

Follow-ups

  • Per-client SSE cost. Each /events connection re-opens and tails the ledger every 500 ms. For a handful of operators this is fine; if the dashboard fans out to many viewers, a shared in-process tail with a broadcast buffer would cut redundant file reads.
  • Loopback operator bypass. Inherited from Sprint 4's threat model: the dashboard relies on peerIsLocal for operator authority. Sprint 6's hardening pass should replace the loopback-TCP bypass with a Unix-domain socket or explicit operator session.
  • Drag-priority within a lane. The board sets the priority tier; it does not order tasks within a tier. If operators want a strict FIFO-within-tier order, that needs a secondary ordering field the projection does not track yet.
  • Alert acknowledgement is client-local. A acks in the browser session only; a refresh clears it. A durable ack would need an ack task_event — deferred until operators ask for cross-session persistence.

SSE additions are backward-compatible: a client that ignores event: ledger frames still gets the chat message stream. The 30 s fallback polls keep the dashboard correct even if the stream drops.
