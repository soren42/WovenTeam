Detailed Technical Status

Policy Module

  • New include/wt_policy.h + src/wt_policy.c expose wtPolicyEvaluateTaskPackage(), a pure function over WtPolicyInput. Caller prepares projection totals (initiative + model-family allocations, daily/monthly windows, agent/initiative active counts); evaluator returns {allowed, reason, message}.
  • Reason allowlist (stable): vendor_blocked, model_agent_mismatch, initiative_budget, model_family_budget, daily_budget, monthly_budget, capacity_agent, capacity_initiative.
  • wtPolicyVendorIsBlocked() CSV-walks blockedVendors with whitespace tolerance so "deepseek, xai" parses identically to "deepseek,xai".

Config Additions

  • blockedVendors — CSV of vendor prefixes. Defaults to deepseek to preserve the existing trust-boundary decision documented in the audit log.
  • tokenBudgetPerInitiative — cap on the sum of budget.maxTokens for active task packages within a single initiative. 0 disables.
  • tokenBudgetPerModelFamily — same idea but grouped by modelId family (prefix before /).
  • All three are settable via key/value config file, WT_* environment variables, and POST /api/config.

Projection Helpers

  • wtTaskProjectionAllocatedTokensForInitiative() + wtTaskProjectionAllocatedTokensForModelFamily() sum max_tokens over non-terminal tasks. Family query uses a LIKE '/%' prefix so openai does not accidentally match openai-experimental.
  • Audit export reuses the existing tasks + task_events SQLite tables and folds in raw JSONL scans for policy decisions + usage events (which are not projected).
  • The audit endpoint matches policy decisions either by taskId IN (initiative_tasks) or by the denial record's own initiativeId — that latter path catches denials for tasks that were rejected before ever reaching the projection.

CLI + Console

  • bin/wt-task audit INIT [--out FILE] wraps GET /api/initiative-audit.
  • bin/wt-task create and request default modelId to empty so the daemon auto-defaults to a family matching the routed (or pinned) agent. Removes the OpenAI default that tripped the new model_agent_mismatch rule.
  • Settings panel adds blocked-vendors + per-initiative + per-family inputs.
  • Composer budget pressure note lists active policy levers (blocked: deepseek, per-initiative cap 10.0M, etc.) so the operator sees what will trip before EXECUTE.
  • Dispatch errors render the evaluator's classified reason inline rather than "dispatch failed".
  • Each initiative card has an Audit button that opens the JSON report in a new tab; mirrors the CLI.

Evaluation Flow

Both /api/task-package and /api/task-request route through the same evaluator. Order matters: most-specific reason wins so the recorded reason code points the operator at the right knob.

Audit Export

The combined audit fits into one response so an operator can hand a single JSON file to compliance or a downstream dashboard. Empty initiatives return HTTP 404.

Follow-ups Into Sprint 6 (Launch Rehearsal)

  • Sprint 6 should drive an end-to-end rehearsal that exercises the policy evaluator against real CLI adapters — at least one accepted artifact, one budgeted denial, and one operator reclaim.
  • The audit export is currently a single 1 MiB buffer. Initiatives with many hundreds of events will eventually need pagination; revisit during Sprint 6 load testing.
  • Model-family budget enforcement only considers maxTokens allocations on task packages, not actual usage events. A future enhancement could let the operator pick between allocation-based or actual-usage-based caps.
  • Topology and command-palette console views remain disabled. Sprint 6 documentation should explicitly bound their scope so they do not block the launch report.

No new operational risks. The evaluator is additive: defaults preserve prior behavior (only deepseek blocked, per-initiative and per-family budgets at 0). Old ledgers without policy_decision records project as empty arrays and the UI degrades gracefully.
