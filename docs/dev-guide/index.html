1. Architecture overview

WovenTeam is C + Bash with a deliberately small, durable core. The source of truth is an append-only JSONL ledger; a rebuildable SQLite projection is derived from it for fast reads. The ledger is authoritative and the projection is disposable — it can be dropped and rebuilt from the ledger at any time. Preserving this property is the single most important architectural invariant.

2. Data model & schemas

Records are newline-delimited JSON, one per line, each tagged with a schema. The authoritative definitions live in docs/api/. Principal schemas:

SchemaPurpose
woventeam.task_package.v0.1A task: role, agent, tool policy, budget, optional autonomy grant.
woventeam.task_event.v0.1Lifecycle events: status, lease, reclaim, artifact, routing, priority, note.
woventeam.task_request.v0.1A manager's request to spawn a sub-task.
woventeam.heartbeat.v0.1 / milestone.v0.1Agent liveness; progress markers.
woventeam.kill_event.v0.1Operator cancel / autonomy revoke, polled by the agent wait loop.
woventeam.autonomy_event.v0.1Grant observation, adapter invocation, elevation, exit, revoke.
woventeam.deliverable.v0.1 / secret_scan.v0.1Shipped artifact manifest (sha256, mode); pre-ship scan result.
woventeam.policy_decision.v0.1Denials with stable reason codes.
woventeam.host_event.v0.1 / auth_token.v0.1Remote host capabilities; bearer token issuance (SHA-256 hashed).

The projection (src/wt_task_projection.c) reads the ledger and materializes tables (tasks, task_events, heartbeats, milestones, kill_events, deliverables, secret_scans). Every projection rebuild drops and recreates these tables, so adding a schema means: (1) define it, (2) append it from a store helper, (3) project it (with a matching DROP in the rebuild).

3. The daemon (wt-roomd)

src/wt_roomd.c is the HTTP daemon. It serves the console (web/), exposes the JSON API, owns the projection, enforces policy, and streams events. It forks a handler per connection; long-lived SSE clients (the dashboard's /events stream) each hold a forked process for up to an hour.

Endpoints

Reads: /api/status, /api/task-summaries, /api/task-detail, /api/initiatives, /api/initiative-audit, /api/hosts. Mutations (operator-gated): /api/task-package, /api/task-request, /api/task-artifact, /api/task-deliverable, /api/task-priority, /api/task-reroute, /api/task-note, /api/task-cancel, /api/autonomy-revoke, /api/auth-token.

SSE

The /events stream emits chat message frames and typed ledger frames (one per new ledger record, tailed by byte offset). The console maps each ledger frame's schema to a debounced refresh of the affected view — event-driven instead of polling.

Shell-out gotcha. The daemon installs SIGCHLD=SIG_IGN, which makes bare popen()/system() return -1 even on success. Use the systemReliable() wrapper (restores SIG_DFL around the call) for any shell-out from the daemon.

4. Agents & adapters

src/wt_agent.c is the worker loop. It claims a leased task from the ledger, emits a heartbeat, and dispatches to an adapter (or the stub path). The wait loop (waitForAdapterChild) owns lease renewal, cancel polling, and SIGTERM/SIGKILL of the adapter process group.

4.1 Adapter contract

An adapter maps an agent to a vendor-CLI invocation in an isolated per-task workspace (runtimeRootPath/<taskId>/), capturing the model's output as result.md and recording stdout/stderr. The full contract is in docs/adapters/adapter-contract-v0.1.md. The shipped adapters:

AgentCLI invocation (non-elevated → elevated)
chatgptcodex exec … --sandbox workspace-write → … --dangerously-bypass-approvals-and-sandbox
claudeclaude --bare --print → … --dangerously-skip-permissions
geminigemini --approval-mode plan -p → … --approval-mode yolo
Two lessons baked into v1.0.1. Adapter child processes must redirect stdin from /dev/null (else codex exec blocks reading stdin), and adapter flags must match the installed CLI surface (the codex adapter shipped an invalid --ask-for-approval flag in v1.0). Verify any new adapter against its real CLI.

4.2 Adding an adapter

  1. Add an eligibility predicate (see isCodexEligibleTask / isCliArtifactEligibleTask) keyed on the agent + tool profile + config switch.
  2. Add the fork/exec body: setpgid(0,0), redirect stdin from /dev/null, redirect stdout/stderr to the workspace logs, build the argv against the real CLI, write the result to result.md.
  3. Wire config switches (enable*Adapter, *Mode, *Command).
  4. Add an integration test using a fake CLI binary (see tests/integration/wt-codex-adapter.sh) plus assertions on the emitted flags — include negative checks for the elevation gating.

5. Roles

Roles are a three-layer construct: behavior (a ROLE.md persona), routing (role YAML mapping a role to a default agent), and capability (model YAML + a suitability map). The role_id must match the suitability key. The spawn hierarchy is enforced in roleMaySpawn() in the daemon: program_manager → architect/PM roles; project_manager → worker roles. To add a role, define its three layers and (if it should be spawnable) add it to the appropriate allow-list.

6. Autonomy & policy

A task may carry an autonomyGrant (scope, ttlSeconds, maxWallClockSeconds, network, credentialClass, requiresCleanWorktree, revokedAtUnixMs). The agent resolves the effective level (per-task → per-initiative → per-agent config → tool-profile default). The policy evaluator (src/wt_policy.c) classifies failures with stable reason codes — autonomy_required, autonomy_expired, autonomy_revoked, plus budget/capacity/vendor codes — and records each denial as a policy_decision. Elevation is fork-time admission control: once the adapter runs elevated, the daemon has no per-call visibility into it.

7. Deliverables pipeline

src/wt_deliverable.c ships an accepted artifact through copy / tarball / branch / pull-request modes. It computes a SHA-256 (via sha256sum through systemReliable()), runs the configured pre-ship secret scan (POSIX regex per pattern), and appends a deliverable manifest plus a secret_scan record. Branch and PR modes refuse on a scan match (secret_scan_block); copy and tarball record but proceed. Supersede chains preserve the predecessor's id + checksum.

8. Remote execution & security

Agents can run on additional hosts: wt-agent --remote URL --bearer TOKEN --host HOST (HTTPS poll, bearer-authenticated) or operator-launched --ssh HOST. A host capability registry advertises which adapters/profiles each host supports; executionHost-pinned tasks route only to matching hosts, rejecting mismatches with capability_unmet.

Mutating endpoints require a local operator session (a urandom token in a 0600 file, handed to loopback callers) or a bearer token issued to the operator role. Bearer tokens are stored SHA-256-hashed in the ledger, never raw. The threat model (docs/security/threat-model.html) documents the single-tenant intranet assumptions and the "local host = operator" residual; cross-network use needs mTLS.

9. Build, test, release

make            # build wt-roomd, wt-agent, wt-task helpers
make test       # full integration suite (24 tests + smoke)
make dist-tarball   # release archive: dist/woventeam-<commit>.tar.gz
Build hygiene. Verify with make clean && make, not an incremental build — incremental builds skip unchanged translation units and have hidden real -Wformat-truncation warnings (one of which could emit malformed JSON). A non-clean compile is a review blocker.

Integration tests live in tests/integration/ and are executable specifications: each spins up a daemon on a random port against a temp ledger and asserts behavior. Adapter tests use fake CLI binaries so they run offline and deterministically.

Releasing

Tag annotated from a green commit, build the tarball from the tag (git archive --prefix=WovenTeam-X.Y.Z/), publish a GitHub Release with the tarball + a .sha256, and mark it Latest.

10. Contributing

  • Workflow: branch → PR → squash-merge → delete branch. Direct-to-main only for an immediate hotfix.
  • No Node / server-side JS. Browser JS is fine; Python is acceptable. Documentation and reports are self-contained HTML; the GitHub wiki is the one Markdown exception.
  • Tests: add or update an integration test for any behavior change; keep make test green on a clean build.
  • Reviewing a peer's C: always do a clean build; assert the corrected behavior with negative checks where gating matters.

11. Post-v1 roadmap

Not yet implemented. The items below are planned for after v1 and are not present in v1.0.x. They are recorded so contributors can see the intended direction.
  • Credentialed autonomous demonstration (worker steps as live model inference end-to-end).
  • Local operator transport: replace the loopback-issued shared session secret with a Unix-domain socket or per-operator session.
  • Per-client SSE fan-out: a shared in-process tail + broadcast instead of a forked handler per client.
  • Cross-namespace remote execution test (the current acceptance test runs over loopback where unprivileged netns is unavailable).
  • Richer in-process job visualization; an open plugin architecture; web-based role and model editors; real-world (provider-polled) token/limit tracking; higher-performance infrastructure (MQ, MySQL) preserving the rebuildable-ledger invariant.
