Phase 0 Agent Access And Workload Roadmap

Generated 2026-05-16 UTC from live inspection of xenon and /woventeam/repos/WovenTeam.

Are agents running on xenon? Yes.

wt-roomd, wt-agent@claude, wt-agent@chatgpt, and wt-agent@gemini are active systemd services on host xenon.

Can agents use harnesses today? No, not through WovenTeam.

Some harness CLIs are installed, but the current native agents are stub responders and have no adapter execution path.

Can role/task/agent packages be added? Yes, feasible.

The path is an orchestrator layer over the existing room: task schema, durable task state, role routing, adapter runners, permissions, and context packaging.

3. Can the common conversation evolve into role + task + agent packages?

Yes. The current room is a good transport and audit substrate, but it is not yet an orchestrator. It needs a task package schema, task ledger, assignment endpoints, routing rules, and agent execution contracts.

Feasibility

The project path is feasible if the next phase keeps the room as the shared conversation layer and adds a separate execution plane. The room should carry task events and human-readable summaries; a task ledger should carry authoritative task state.

Roadmap

PhaseGoalDeliverablesExit criteria
0.1 Capability Inventory Make host and harness reality visible. wt-harness-check, JSON capability report, model-profile drift report. Report says which agents are installed, authenticated, launchable, writable, and supported.
0.2 Task Package Schema Represent role + task + agent as a durable package. woventeam.task_package.v1, task JSONL or SQLite ledger, validation tests. CEO can create a structured task package and see it in the room and ledger.
0.3 Router Resolve roles to models and agent instances. Role/model profile loader, suitability filter, blocked-vendor filter, missing-harness filter. Router selects an available model or explains why no assignment is possible.
0.4 Adapter Runner Run meaningful work through one safe harness. Codex-first adapter with stdin prompt, bounded stdout/stderr, timeout, workspace path, audit event. One task executes in a controlled workspace and writes a task result without manual shell use.
0.5 Claude Adapter Decision Define an approved Claude execution path. Design note choosing human-mediated, API-backed, or other approved mode; no implicit automation. Claude tasks are either explicitly supported by policy or routed elsewhere.
0.6 Manager Spawn Flow Allow PM/PgM agents to create sub-agent packages. /api/task-package, task.request, task.assign, dependency and gate states. A PM task can spawn a backend or tester subtask with bounded context and permissions.
0.7 Operations Hardening Make boot and runtime reliable. Per-agent runtime homes, systemd templates, journald correlation IDs, crash/retry policy. Agents survive reboot and do not lose in-progress task state.
0.8 Dashboard Expose task state to CEO and managers. Task list, filters, status badges, role/model assignment display, result links. CEO can see pending, running, blocked, complete, and failed tasks.

Project Plan

Sprint 1: Reality And Schema

  1. Add docs/api/task-package-v0.1.json with the package schema.
  2. Add wt-harness-check to inspect installed CLIs, version, launchability, service sandbox write constraints, and model-profile drift.
  3. Add tests for missing Gemini/Qwen/OpenAI CLI mismatch and installed Claude/Codex/Ollama visibility.
  4. Add a task ledger in JSONL first, or SQLite if task querying becomes immediately necessary.

Sprint 2: Assignment Path

  1. Add wt-task CLI: create, list, show, assign, update-status.
  2. Add POST /api/task-package and GET /api/tasks to wt-roomd.
  3. Emit room-visible task.assign, task.status, and task.result events.
  4. Teach wt-agent to poll task packages by assigned agent and status.

Sprint 3: First Meaningful Work Adapter

  1. Implement one adapter runner, preferably Codex-first because the CLI is installed and already being used on this host.
  2. Use per-task workspaces under /woventeam/runtime/tasks/<taskId>.
  3. Use bounded input/output, timeout, exit-code capture, and artifact manifest writing.
  4. Keep tool permissions conservative: repo read/write only in a branch workspace; no live system changes.

Sprint 4: Manager-Driven Subtasks

  1. Implement task.request messages for PM/PgM roles.
  2. Add policy checks: which roles may spawn which other roles, max concurrency, and required approval gates.
  3. Add dependency handling and blocked-state propagation.
  4. Add smoke test: CEO creates initiative, PM creates two subtasks, worker completes one, tester reviews result.

Systemd And Runtime Changes Needed

  • Replace ProtectHome=read-only for harness runners with dedicated writable runtime homes, such as /woventeam/runtime/homes/<agent>, or bind-mount only the required credential files read-only plus a writable cache/state directory.
  • Keep wt-roomd and task ledger small and native; do not embed vendor SDKs in the daemon.
  • Run adapters as child processes with explicit argv arrays, not shell-concatenated commands.
  • Capture stdout/stderr separately, cap output, and store large artifacts as files referenced from task results.
  • Record every adapter launch in the task ledger with command path, argv summary, timeout, exit status, output hash, and workspace path.
  • For Gemini adapters, set an explicit trusted-workspace policy for headless execution instead of relying on an interactive trust prompt.

Bottom Line

The current common chat system has proven durable shared state and live stub participants on xenon. It is not yet a work-capable multi-agent system. The path from here to role/task/agent packages is feasible and should be built as a staged orchestration layer rather than by letting chat messages execute commands directly.
