1. Introduction

WovenTeam is a multi-agent orchestration workbench. You — the operator — open initiatives, file tasks, and route them to agents (Claude, ChatGPT/Codex, Gemini). Agents claim leased tasks and either run as stubs or invoke a real model adapter to produce work product, which you review, promote, and ship as a deliverable. Every action is recorded in an append-only ledger you can audit.

The system is built in C and Bash around a small durable core: an append-only JSONL ledger as the source of truth and a rebuildable SQLite projection for fast reads. It is designed for a single operator on a trusted intranet.

What v1.0.2 is. A working control plane and operator workbench: a daemon, a CLI, a real-time dashboard, a deliverables pipeline, remote execution, and release hardening. Real adapter-backed execution is implemented and proven, but disabled by default — see chapter 8.

The pieces

ComponentRole
wt-roomdThe daemon — serves the console + JSON API, owns the projection, enforces policy, streams events, records audit.
wt-agentA worker loop per agent — claims tasks and runs them (stub or real adapter).
wt-taskThe operator CLI — everything you do from the terminal.
ConsoleThe browser dashboard at http://127.0.0.1:8787/.

2. Installation

WovenTeam runs on a GNU/Linux host. Install the prerequisites from your package manager: cc, make, the sqlite3 development headers, curl, jq, tar, gzip, and git. Real adapter-backed workers additionally need the relevant vendor CLI (codex, claude, and/or gemini) authenticated on the host.

From the release tarball (recommended)

The current release is v1.0.2.

cd ~
curl -fsSLO https://github.com/soren42/WovenTeam/releases/download/v1.0.2/woventeam-1.0.2.tar.gz
curl -fsSLO https://github.com/soren42/WovenTeam/releases/download/v1.0.2/woventeam-1.0.2.tar.gz.sha256
sha256sum -c woventeam-1.0.2.tar.gz.sha256
tar -xzf woventeam-1.0.2.tar.gz
cd WovenTeam-1.0.2
make clean && make && make test

make test runs the full integration suite (24 tests + smoke); all should pass on a clean host. A cold install completes in well under 30 minutes when prerequisites are present.

Configure

cp config/woventeam-phase0.conf.example config/woventeam-phase0.conf

Key settings: httpPort (default 8787), tokenDailyBudget/tokenMonthlyBudget, deliverableRoot, the enable*Adapter / *Mode adapter switches, and secretScanPatternsFile.

Run as a service (optional)

For an unattended host, install the systemd units: make install-roomd-service and make install-agent-services. These run one daemon and one agent set; do not also launch agents by hand, or two sets will race on the same ledger.

3. Quick start

Start the daemon, then the agents in separate terminals:

./build/wt-roomd --config config/woventeam-phase0.conf
# in other terminals:
./build/wt-agent --agent claude  --loop --config config/woventeam-phase0.conf
./build/wt-agent --agent chatgpt --loop --config config/woventeam-phase0.conf
./build/wt-agent --agent gemini  --loop --config config/woventeam-phase0.conf

Open the console at http://127.0.0.1:8787/. File your first task from the CLI:

export WT_ROOM_URL=http://127.0.0.1:8787
./bin/wt-task create --title "Hello WovenTeam" --body "Confirm the assignment path." \
  --role backend_dev --agent chatgpt --max-tokens 1000
./bin/wt-task list

Watch it move through the dashboard, then promote and ship its artifact (chapter 7).

4. Concepts

4.1 Initiatives & tasks

An initiative is a unit of work with a budget and a set of tasks. A task is a single assignment with a role, an assigned agent, a tool profile, and an optional autonomy grant. Tasks can request sub-tasks, forming a tree under the initiative.

4.2 Roles & the spawn hierarchy

Every task carries a role, and the daemon enforces who may spawn whom:

A program_manager may spawn project_manager and architect roles only; a project_manager spawns the worker roles (backend_dev, tester, technical_writer, etc.). Attempting to skip a layer is rejected with role spawn policy denied.

4.3 Task lifecycle

Tasks move through states recorded as ledger events: queued → assigned → leased → running → complete (or failed, blocked, cancelled, closed). An artifact a task produces has its own decision states: draft → reviewed → accepted (or rejected, superseded). A lease that expires while running surfaces as a stuck-task alert and can be reclaimed.

5. Operating with the CLI

The wt-task CLI talks to the daemon over HTTP. Point it at your daemon with WT_ROOM_URL (default http://127.0.0.1:8787). The complete command surface is in the wiki CLI Reference; the common flows:

Open work

wt-task initiative create --id init_demo --title "Demo" --body "..." --role program_manager --agent claude
wt-task request --parent <PGM_TASK> --by-role program_manager --role project_manager --title "Plan it" --body "..." --agent claude
wt-task request --parent <PM_TASK>  --by-role project_manager --role backend_dev --title "Build it" --body "..." --agent chatgpt

Inspect

wt-task list
wt-task show TASK_ID
wt-task initiative show INIT

Steer

wt-task reclaim TASK_ID            # release a stuck lease back to the queue
wt-task cancel-task TASK_ID        # kill-tree a running adapter
wt-task autonomy revoke TASK_ID    # revoke an autonomy grant

6. Operating with the dashboard

The console at http://127.0.0.1:8787/ mirrors the CLI and updates in real time over a server-sent-events stream — task state changes appear within ~2 seconds of the ledger append. It shows a glanceable summary bar (initiatives, in-flight tasks, alerts, budget pressure, oldest stuck task, autonomy grants), the task table, a priority-lane board you can drag tasks between, per-agent heartbeats, recent milestones, and a deliverables panel.

In-flight steering

The task detail panel lets you change priority, reroute to another agent, add an operator note, cancel, and revoke autonomy — each lands as a ledger event.

7. Reviewing & shipping deliverables

When an agent produces an artifact, you review and accept it, then ship it out of the per-task workspace into a stable destination.

wt-task artifact promote TASK_ID --path result.md --reviewer ceo --notes "ship it"
wt-task artifact ship TASK_ID --mode copy

Packaging modes

ModeWhat it does
copyCopies the file into deliverableRoot/<initiativeId>/. Default.
tarballGzipped tarball under the deliverable root.
branchCommits + pushes to a deliverables branch in a project repo (--repo PATH).
pull-requestBranch + opens a PR via gh; requires explicit --yes.

Every ship records a deliverable manifest with a SHA-256 checksum and a secret_scan result. Branch and pull-request modes refuse to ship if the pre-ship secret scan matches a configured pattern; copy and tarball record the scan but proceed.

8. Autonomy & real adapters

Default is stub. Out of the box, agents run as stubs (enableCodexAdapter=0, chatgptMode=stub): they exercise the assignment path without invoking a model. To run live models you enable the adapter and supply credentials.

Enabling a real adapter

In your config, enable the adapter and point it at an authenticated vendor CLI, e.g. for ChatGPT/Codex:

enableCodexAdapter=1
chatgptMode=adapter
gptCommand=codex

The agent then invokes the real CLI in an isolated per-task workspace and captures the result as the artifact.

Autonomy grants

An autonomous task runs the adapter elevated (fully non-interactive). It requires a bounded grant — a scope, a time-to-live, and limits — supplied when you create the task:

wt-task create --title "..." --body "..." --role backend_dev --agent chatgpt \
  --tool-profile repo_branch --autonomy-level autonomous --autonomy-ttl 3600 --autonomy-scope adapter

Grants are revocable at any time with wt-task autonomy revoke TASK_ID, which records the revocation and tears down a running adapter. The policy evaluator rejects autonomous tasks lacking a valid grant (autonomy_required), with expired grants (autonomy_expired), or after revocation (autonomy_revoked).

Trust boundary. Once an adapter is launched with elevated permissions, WovenTeam relinquishes per-action visibility into that adapter. Autonomy bounding is fork-time admission control, not a per-call sandbox.

9. Audit & reporting

Everything is recorded in the ledger. Two operator views:

wt-task audit INIT                      # full initiative audit as JSON
wt-task initiative report INIT --out report.html   # standalone HTML report

The HTML report renders the initiative's tasks, deliverables (with checksums), secret scans, policy denials, token spend, and a full event timeline as a single self-contained file you can archive or share. All values are HTML-escaped, so a report is safe to open even when the ledger contains hostile-looking data.

10. Troubleshooting

SymptomCause / fix
Task created but rejected with daily_budgetThe initiative's tasks exceeded tokenDailyBudget. Raise the budget or lower --max-tokens.
role spawn policy deniedYou tried to skip a layer (e.g. a program_manager spawning a worker). Go through a project_manager.
Agent idle, tasks not progressingCheck only one agent set is running (systemd or manual, not both). Two sets race on the same ledger.
Real adapter task fails immediatelyConfirm the vendor CLI is on PATH and authenticated, and the adapter is enabled in config.
Operator endpoint returns operator_auth_requiredMutating endpoints require a local-session or bearer token. The browser console and local CLI obtain a session automatically.
